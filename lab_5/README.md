Застосування нормалізації: перетворення таблиць у вищі нормальні форми (до 3НФ) для усунення часткових та транзитивних залежностей.

## 1 Початкова (ненормалізована) схема бази даних
Для аналізу процесу нормалізації розглянемо початкову таблицю `User_Purchases_Denormalized`, яка містить усю інформацію про користувачів, їхні замовлення та ігри в єдиній структурі.

Ця таблиця має серйозні недоліки, такі як дублювання даних та аномалії оновлення/вставки/видалення.

Початкова таблиця: `User_Purchases_Denormalized`
* `OrderID` (Ключ)
* `GameID` (Ключ)
* `OrderDate`
* `UserID`
* `Nickname`
* `Email`
* `GameTitle`
* `GameDescription`
* `GameGenres` (містить значення через кому, наприклад: "Action, RPG, Open World")
* `PriceAtPurchase`

Поточна нормальна форма: 0NF (Ненормалізована), оскільки наявні багатозначні атрибути (порушення 1NF).

---

## 2 Функціональні залежності (ФЗ)
Для початкової таблиці визначено наступний мінімальний набір функціональних залежностей:
1. `OrderID`, `GameID` → `PriceAtPurchase` (Ціна покупки залежить від конкретної гри у конкретному замовленні)
2. `OrderID` → `OrderDate`, `UserID`, `Nickname`,`Email` (Дані про замовлення та клієнта залежать тільки від номера замовлення)
3. `GameID` → `GameTitle`, `GameDescription`, `GameGenres` (Дані про гру залежать тільки від ID гри)
4. `UserID` → `Nickname`, `Email` (Дані користувача однозначно визначаються його ID)

---

## 3 Перехід до Першої нормальної форми (1NF)
**Правило 1NF:** Усі атрибути мають бути атомарними. Не повинно бути повторюваних груп або списків у межах однієї колонки.
**Проблема:** Поле `GameGenres` порушує 1NF, оскільки містить кілька жанрів через кому. Це створює аномалії пошуку (складно знайти всі ігри жанру "RPG") та оновлення.
**Рішення:** Виносимо жанри в окремі структури. Створюємо таблиці `Genres` та проміжну таблицю `GameGenres` для зв'язку "багато-до-багатьох".
**Схема в 1NF:**
* `User_Purchases` (OrderID, GameID, OrderDate, UserID, Nickname, Email, GameTitle, GameDescription, PriceAtPurchase)
* `Genres` (GenreID, GenreName)
* `GameGenres` (GameID, GenreID)

Чому це усуває аномалію: Тепер кожен жанр є атомарним значенням у власному рядку, що дозволяє легко додавати нові жанри та виконувати пошук без парсингу рядків.

---

## 4 Перехід до Другої нормальної форми (2NF)
**Правило 2NF:** Таблиця знаходиться в 1NF і не має часткових залежностей (кожен неключовий атрибут повинен залежати від усього складеного ключа, а не від його частини).
**Проблема:** Таблиця `User_Purchases` має складений первинний ключ `(OrderID, GameID)`. Проте:
* Атрибути `OrderDate`, `UserID`, `Nickname`, `Email` залежать лише від `OrderID` (ФЗ №2).
* Атрибути `GameTitle`, `GameDescription` залежать лише від   `GameID` (ФЗ №3).
Це призводить до жахливого дублювання даних: якщо гру купили 1000 разів, її назва і опис продубльовані 1000 разів. Якщо ми змінимо опис гри, нам доведеться оновлювати всі 1000 записів (аномалія оновлення).

**Рішення:** Декомпозиція таблиці. Розбиваємо її на сутності Замовлення, Ігри та Елементи замовлення (де залишається атрибут `PriceAtPurchase`, який дійсно залежить від обох ключів).

**Схема в 2NF:**
* `Games` (GameID, GameTitle, GameDescription)
* `Orders_Temp` (OrderID, OrderDate, UserID, Nickname, Email)
* `OrderItems` (OrderItemID, OrderID [FK], GameID [FK], PriceAtPurchase)

Чому це усуває аномалію: Інформація про гру тепер зберігається лише в одному місці (таблиця `Games`). Зміна назви гри відбувається лише в одному рядку і автоматично відображається для всіх замовлень.

---

## 5 Перехід до Третьої нормальної форми (3NF)
**Правило 3NF:** Таблиця знаходиться в 2NF і не має транзитивних залежностей (неключові атрибути не повинні залежати від інших неключових атрибутів).

**Проблема:** У таблиці `Orders_Temp` неключові атрибути `Nickname` та `Email` залежать від атрибута `UserID` (ФЗ №4), який сам не є первинним ключем цієї таблиці (транзитивна залежність OrderID → UserID → Email). Якщо користувач змінить email, доведеться шукати всі його минулі замовлення і оновлювати їх (аномалія оновлення).

**Рішення:** Виносимо дані користувача в окрему таблицю Users. В таблиці Orders залишаємо лише UserID як зовнішній ключ.

**Схема в 3NF (Фінальна):**
* `Users` (UserID, Nickname, Email)
* `Orders` (OrderID, OrderDate, UserID [FK])
* `Games` (GameID, GameTitle, GameDescription)
* `OrderItems` (OrderItemID, OrderID [FK], GameID [FK], PriceAtPurchase)
* `Genres` (GenreID, GenreName)
* `GameGenres` (GameID [FK], GenreID [FK])

*Чому це усуває аномалію:* Дані користувача зберігаються одноразово в таблиці `Users`. Оновлення пошти чи нікнейму не зачіпає таблицю замовлень. Усі таблиці тепер знаходяться в 3NF, оскільки кожен атрибут залежить від ключа, всього ключа і ні від чого, крім ключа.

## 6 Фінальний DDL-скрипт (SQL) та ER-діаграма
```SQL
CREATE TABLE Users (
    UserID SERIAL PRIMARY KEY,
    Nickname VARCHAR(100) NOT NULL,
    Email VARCHAR(150) NOT NULL UNIQUE,
    Phone VARCHAR(20),
    RegDate TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE Games (
    GameID SERIAL PRIMARY KEY,
    Title VARCHAR(200) NOT NULL,
    Description TEXT,
    ReleaseDate DATE,
    AgeRating VARCHAR(10),
    BasePrice DECIMAL(10, 2) DEFAULT 0.0 CHECK (BasePrice >= 0)
);

CREATE TABLE Orders (
    OrderID SERIAL PRIMARY KEY,
    UserID INTEGER NOT NULL REFERENCES Users(UserID) ON DELETE CASCADE,
    OrderDate TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    TotalAmount DECIMAL(10, 2) NOT NULL CHECK (TotalAmount >= 0)
);

CREATE TABLE OrderItems (
    OrderItemID SERIAL PRIMARY KEY,
    OrderID INTEGER NOT NULL REFERENCES Orders(OrderID) ON DELETE CASCADE,
    GameID INTEGER NOT NULL REFERENCES Games(GameID),
    PriceAtPurchase DECIMAL(10, 2) NOT NULL CHECK (PriceAtPurchase >= 0)
);

CREATE TABLE Genres (
    GenreID SERIAL PRIMARY KEY,
    GenreName VARCHAR(50) NOT NULL UNIQUE
);
```

## Висновок
У ході роботи початкову ненормалізовану схему бази даних було успішно приведено до Третьої нормальної форми (3НФ). Шляхом послідовного усунення багатозначних атрибутів (1НФ), часткових (2НФ) та транзитивних залежностей (3НФ), дані були розділені на логічні незалежні сутності (`Users`, `Games`, `Orders` тощо). Оновлена структура повністю позбавлена дублювання інформації та аномалій оновлення, що гарантує високу цілісність даних і готовність до подальшого масштабування.
